## TL;DR
「帯域律速ワークロードは DRAM+CXL を最適比でインターリーブした tier に移して実行せよ」
という近年の通説は、CXL メモリ上に置かれたデータをその tier へ移動するコストを見落と
している。本論文はメインメモリハッシュジョイン(PHJ / NPHJ)を題材に、①インターリーブ
tier への移動は移動中のメモリトラフィックが必ず CXL 側に偏り、DRAM への直接移動より
遅いことを示し、②「データの一部(比率 x)だけを CXL→DRAM に移し、DRAM と CXL を並列
アクセスして実行する」戦略の end-to-end 性能モデル(最適 x* = (√(mk)−1)/m)を提案。
実 CXL 1.1 機で、次善戦略(in-CXL 実行)比 PHJ 22.20% / NPHJ 3.59% の実行時間短縮を主張。

## Problem & motivation
- [paper] DRAM スケーリングの壁: bandwidth-per-core / capacity-per-core が頭打ちになり、
  近代 DBMS の性能向上が鈍化 (§1)。CXL は PCIe 上のキャッシュコヒーレントなメモリ拡張で、
  rack-scale pooling と DDR 外の補助帯域を提供する(ただしアクセスレイテンシは高い)(§1, §2.1)。
- [paper] CXL メモリと DRAM を並列アクセスすると DDR + PCIe 両インターフェースの帯域を
  使え、単独をどちらも超える合計帯域が得られる。これを受けて先行研究の多くは、DRAM と
  CXL のページをシステム固有の最適比(≒DRAM 帯域 : CXL 帯域)でインターリーブした新しい
  メモリ tier を作り、ワークロード全体をそこへ移して in-place 実行することを推奨してきた
  (§1, §2.1, Fig. 1)。
- [paper] しかしこの通説は「インターリーブ tier へデータを移動するコスト」を見落としている。
  対象ワークロードは数百 GB 以上に及ぶことが多く、DRAM 容量温存のため CXL メモリに
  置かれるのが一般的な配備形態(tiered / rack-scale メモリ上の大規模 DBMS・DL serving)。
  全量移動のオーバーヘッドは帯域の利得を打ち消し、場合によっては CXL メモリ内で
  そのまま実行するより end-to-end で遅くなる (§1)。
- [paper] 前提注記: インターリーブ tier がワークロード全体を収容できると仮定。できない
  場合は追加のデータ移動でさらに劣化する (p.1, footnote 3)。
- [paper] インターリーブ設定は現状 OS レベルでのみ構成でき、一度設定すると比率は固定で
  全ユーザ空間アプリケーションに共有される (§2.1)。
- [paper] 題材とする PHJ(radix partitioning でキャッシュサイズに分割してから join。
  partition フェーズが支配的)と NPHJ(SMT / out-of-order 実行でキャッシュミスを隠蔽する
  build+probe。probe フェーズが支配的)はいずれも帯域律速で、インターリーブの恩恵を
  受けるはずの代表格 (§2.2)。

## System model & assumptions
- [paper] 実験環境: CXL 1.1 準拠の実機。ホスト DDR5 DRAM 32GB(ピーク 30.01GB/s)、
  CXL type-3 メモリ 64GB(PCIe 5.0 経由 20.95GB/s)。DRAM:CXL = 3:2 のインターリーブで
  最大合計帯域 47.50GB/s (§4.1)。システム最適比は α:β = 60%:40% (Fig. 2 caption)。
- [paper] 入力データは初期状態で CXL メモリに置かれている(DRAM 容量温存のため)という
  配置を前提 (§1, §4.1)。
- [paper] 操作モデル: ハッシュジョインは blocking operator であり、PHJ join フェーズを除く
  全フェーズは「データ移動フェーズ → クエリ実行」という共通手続きに抽象化できる
  (§3.2, Fig. 3)。このモデルはハッシュジョインに限らず、データ移動を伴う帯域律速の
  操作一般に適用可能と主張 (§3, §3.2)。
- [paper] モデルの仮定: 移動スループット p は x に依存しないシステム固有値で p = kr と
  置く(r は in-CXL 実行スループット、k は係数)。実行スループット q = f(x)·r は線形近似
  f(x) = mx + 1(m はシステム固有係数)。合計帯域は DRAM 比率 0→α で単調増加、α 超で
  単調減少という既存研究の観測に基づき、x の定義域を [0, α] に制限 (§3.2)。
  線形は単純化であり実際の関係は非線形かもしれないが、実用上十分な精度と主張 (§3.2, §4.3)。
- [paper] 近未来の CXL メモリ帯域は DRAM を超えないという想定 (§3.1)。将来も k, m は
  1 を上回る水準に留まる見込みと予測 (§4.4)。
- [paper] データ移動の実体はソースからデスティネーションへのメモリページの逐次複製で、
  ソースとデスティネーションへのメモリトラフィックは等量 (§3.1)。
- [inference] モデルに DRAM 容量制約が入っていない。動機部 (§1) は数百 GB 級ワークロードを
  想定するが、x*·|データ| が DRAM 空き容量を超えるケース(まさに動機のシナリオ)の扱いは
  規定されていない。また移動は「複製」なので CXL 側の元コピーは残り、移動分だけ総メモリ
  フットプリントが増える(容量が解放されない)ことになるはずだが、本文に明示は無い。
- [inference] 単一クエリ実行が暗黙の前提。複数クエリが DRAM/CXL 帯域を奪い合う状況や、
  OS 全体で固定共有されるインターリーブ設定 (§2.1) と他アプリの共存は扱われていない。

## Approach
- [paper] **§3.1 移動先分析**: CXL からインターリーブ tier(DRAM:CXL = x:y、y = 100−x)へ
  ページを移動するとき、ソース読み(CXL)= x+y、デスティネーション書き = DRAM に x +
  CXL に y。よって CXL トラフィック = x+2y、DRAM トラフィック = x となり、比
  T(x) = (x+2(100−x))/x = 200/x − 1 (Eq. 1, Fig. 2a)。T(x) は x→100% でもシステム最適比
  β/α に漸近するだけで到達しない。つまりインターリーブ tier への移動中はトラフィックが
  必ず CXL 側に偏り、DRAM 帯域を使い切れない (§3.1)。
- [paper] マイクロベンチマーク (Fig. 2b): 移動スループットはデスティネーション側 DRAM
  比率に対して単調増加。最適比 x=α のインターリーブ tier への移動は中程度のスループット
  しか出ない。結論: 「CXL→DRAM 直接移動」があらゆるインターリーブ tier への移動より
  高スループット (§3.1)。
- [paper] **§3.2 end-to-end 性能モデル**: in-CXL 実行の正規化コスト cost₁ = 1/r (Eq. 2)。
  比率 x を DRAM に移して DRAM/CXL 並列アクセスで実行すると
  cost₂ = x/(kr) + 1/((mx+1)r) (Eq. 3)。利得 g(x) = cost₁/cost₂ = (kmx+k)/(mx²+x+k)
  (Eq. 4) を微分 (Eq. 5) して最大点 x* = (√(mk)−1)/m を導出(x* > α なら α でキャップ)。
  最大期待利得は φ(m,k) = mk/(2√(mk)−1)(mk ≥ 1 のとき。mk < 1 なら移動せず in-CXL
  実行が最善で φ = 1)(Eq. 6)。
- [paper] **§3.3 ハッシュジョインへの適用**:
  - PHJ: partition フェーズの入力テーブルの x* 分を DRAM に移して加速。この x* は
    ピーク帯域の最適インターリーブ比に従う必要はない(従うと過剰な移動コスト)。一方
    partition 出力(マテリアライズ先)は最適比のインターリーブ tier に置く。join フェーズは
    キャッシュサイズのパーティションを 1 回スキャンするだけで、既にインターリーブ tier に
    あるためモデルの適用対象外。partition が PHJ の実行時間を支配するので全体利得は大きい (§3.3)。
  - NPHJ: build / probe 両フェーズに適用可能で、利得は Eq. 6 から直接見積もれる (§3.3)。

## Evaluation
- Setup [paper]: 上記 CXL 1.1 実機 (§4.1)。標準的な合成 equi-join ベンチマーク、タプル
  16B 固定、build 側 16M/64M/256M × probe 側 256M 固定で build:probe 比を変化 (§4.1)。
  入力は初期状態で CXL メモリ。4 戦略を比較: (1) in-CXL 実行、(2) 全量 DRAM 移動 +
  DRAM 実行、(3) 全量を最適比インターリーブ tier へ移動 + そこで実行、(4) モデルが導く
  最適比率だけ部分移動 + DRAM/CXL 並列アクセス実行(提案)。全戦略で中間結果
  (PHJ パーティション、NPHJ ハッシュ表)は最適比インターリーブ tier に置く
  (§4.1, footnote 4)。事前プロファイルで k, m を測定: PHJ は k=2.3534, m=1.0977,
  x*=0.5528, 期待利得 g(x*)=1.1703。NPHJ は k=3.7879, m=0.3641, x*=0.5104,
  g(x*)=1.0214 (Table 1)。
- Headline [paper] (Fig. 4, §4.2):
  - 提案戦略が全アルゴリズム・全 build:probe 比で最良。次善戦略(in-CXL 実行)比で
    PHJ 22.20% / NPHJ 3.59% の実行時間短縮。Table 1 の期待利得とよく一致し、build:probe
    比によらず安定 (§4.2)。
  - 提案を除くと、PHJ / NPHJ とも「全量を高速 tier に移して in-place 実行」より in-CXL
    直接実行の方が速い(移動最小化の便益を裏付け)(§4.2)。
  - 全量移動同士の比較では、DRAM への複製の方がインターリーブ tier への移動より移動
    コストが低い(§3.1 の分析を裏付け)(§4.2)。
  - 配置後の実行フェーズ自体の性能は戦略間でほぼ差なし(全戦略が中間データを
    インターリーブ tier に置くため)(§4.2)。
  - NPHJ の利得が小さい理由: m が小さい。probe のメモリトラフィックはハッシュ表への
    ランダムアクセスが支配的で、ハッシュ表は build 後すでに最適比インターリーブ tier に
    あるため、probe 側データの置き場所の影響が小さい (§4.2)。
  - PHJ は NPHJ より全体に速い(キャッシュミス削減。先行研究と整合)(§4.2)。
- 感度分析 [paper] (§4.3, Fig. 5): partitioning を対象に x を掃引。モデル予測の最適
  x = 0.5528 に対し実測最適は 0.4972、予測利得 17.03% に対し実測最大 24.20%(いずれも
  partitioning フェーズ単体の値。join フェーズには提案を適用しないため Table 1 より高い、
  footnote 5)。実測最適が予測よりやや低いのは、α 近傍で帯域改善が sublinear になり、
  より少ない移動で同等帯域に届くため(線形 f(x) の限界)(§4.3)。x から離れると予測と
  実測は乖離するが傾向は一致し、モデルが x < α を示す点で実用的と総括 (§4.3)。
- 将来予測 [paper] (§4.4, Fig. 6): λ = mk と置くと予測利得は λ に対して単調増加
  (本システムは λ=2.5353 の点、Fig. 6 caption)。CXL/PCIe の帯域向上ロードマップと
  「CXL 帯域 < DRAM 帯域は継続」の想定から、部分移動戦略は次世代 CXL 機でも有効と主張。
  さらにローカル DDR チャネルと独立な追加メモリ帯域を提供する任意のインターコネクト
  (NVIDIA NVLink、NUMA)に方法論が適用可能と position (§4.4)。
- [inference] 評価がカバーしていないもの:
  - データ規模が小さい: 256M × 16B = 1 テーブル約 4GB で、DRAM 32GB に全量が余裕で
    収まる。動機 (§1) の「数百 GB 級・DRAM に入らないから CXL に置く」シナリオ、すなわち
    x* 分すら DRAM に入らない容量制約下の挙動は未評価。
  - 単一プラットフォーム・単一 CXL デバイス(CXL 1.1)のみ。k, m, α はシステム固有で、
    帯域比の異なるデバイス(ASIC 型、スイッチ経由プール等)での線形近似 f(x) の妥当性は
    未検証(§4.4 の将来予測は Eq. 6 の外挿のみ)。
  - 合成 equi-join のみで、キー分布のスキュー、実クエリ(selection / 多結合)、同時実行
    クエリ下の帯域競合は無い。タプルサイズも 16B 固定。
  - k, m は Table 1 でアルゴリズムごとに大きく異なる(k: 2.35 vs 3.79、m: 1.10 vs 0.36)。
    つまり係数は「システム固有」というよりワークロード依存であり、オペレータごとの
    オフラインプロファイルが前提になるが、そのコストや汎化は論じられていない。
  - 移動と実行のオーバーラップ(移動をパイプライン化して partition と並行させる等)は
    戦略として比較されていない(モデルは逐次 2 フェーズを仮定)。
- [question] 数値の内部整合: Table 1 の NPHJ 行は k=3.7879, m=0.3641 だが、これを
  x* = (√(mk)−1)/m に代入すると ≈0.479 となり、記載の x*=0.5104 と合わない(PHJ 行は
  一致)。また Fig. 6 caption の λ=2.5353 は k=2.3534 × m=1.0977 = 2.5833 と一致しない。
  テキスト抽出の文字化けの可能性もあるため、原 PDF の表・図で要再確認。

## Limitations
- Stated [paper]:
  - NPHJ への利得は本質的に小さい(3.59%)。probe のトラフィックがインターリーブ tier 上の
    ハッシュ表へのランダムアクセスに支配されるため (§4.2)。
  - PHJ の join フェーズにはモデルが適用できない(移動不要のキャッシュ内 join)(§3.3)。
  - f(x) の線形近似は単純化。α 近傍の sublinear 性により実測最適 x はモデル予測より低い
    (§3.2, §4.3)。
  - インターリーブ tier がワークロード全体を収容できることが前提 (p.1, footnote 3)。
- Inferred [inference]:
  - モデルに DRAM 容量制約が無い。「CXL に置くのは DRAM が足りないから」という動機の
    シナリオでは x* 分の DRAM 確保自体が非自明で、複製による一時的フットプリント増も
    容量圧力を高める。評価データは DRAM に全量収まる規模のみ。
  - モデル適用にはワークロード毎の事前プロファイル(r, p, f(x) の傾き)が必要で、DBMS の
    オプティマイザ / 実行エンジンへの統合方法(オンライン推定、プラン選択との連携)は
    示されていない。CIDR ビジョンペーパーとしての位置付けと整合的だが、実システム化の
    ギャップは大きい。
  - 中間結果は全戦略で「最適比インターリーブ tier」に置く設計 (§4.1, footnote 4) だが、
    OS レベルで固定・全アプリ共有の interleave 設定 (§2.1) の下で、入力用の DRAM 直接
    確保と出力用インターリーブ tier をどう同居させるか(NUMA ポリシーの使い分け等)の
    実装詳細は本文からは読み取れない。

## Idea seeds
- [inference] モデルの容量制約付き拡張: DRAM 空き容量 C の下で x = min(x*, C/|data|) と
  クリップした場合の利得低下を定式化し、buffer pool と部分移動が DRAM を奪い合う
  larger-than-memory 設定での最適配分を解く。最初の検証: 公開コード
  (https://github.com/fukien/hashjoins-meet-cxl, p.1 footnote 1)で DRAM を cgroup 等で
  絞り、x 掃引カーブ (Fig. 5 相当) が容量壁でどう折れるかを実測する。
- [inference] 移動と実行のパイプライン化: モデルは「移動 → 実行」の逐次 2 フェーズを
  仮定するが (§3.2, Fig. 3)、radix partitioning はチャンク単位で進むため、移動をチャンク
  粒度で partition と並行させれば cost₂ の x/p 項を部分的に隠蔽できるはず。逐次最適 x* と
  パイプライン下の最適 x* は異なる値になると予想される。検証: 公開コードに async 移動
  スレッドを足して 4 戦略 + パイプライン戦略の 5 者比較。
- [inference] k, m のオンライン推定: Table 1 が示す通り係数はアルゴリズム依存なので、
  クエリオプティマイザが物理プラン選択時に x を決めるには実行時プロファイルが要る。
  クエリ冒頭の数百 ms で移動スループットと f(x) の傾きをマイクロプローブし、残りの
  データで x を適応制御する「self-tuning 部分移動」はビジョンの自然な次の一歩
  (§5 の future work は他の帯域律速オペレータ・多結合クエリ・他のメモリ拡張技術への
  一般化までで、オンライン化には触れていない)。
- [question] 同時実行クエリが DRAM / CXL / インターリーブ tier の帯域を共有するとき、
  各クエリの x* は他クエリの配置に依存するはず(ゲーム論的/大域最適化問題になる)。
  複数ジョイン同時実行で各自が単独最適 x* を使うと合計スループットはどれだけ劣化するか。
  検証: 公開コードを 2〜4 インスタンス並走させて単独最適 x* の劣化を測る。
